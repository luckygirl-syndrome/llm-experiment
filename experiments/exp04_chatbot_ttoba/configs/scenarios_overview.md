## 시나리오 구성 원칙

### 왜 이 조합인가

또바가 제대로 작동하는지 확인하려면 두 가지를 검증해야 한다.

1. **점수가 다를 때 또바의 방향이 달라지는가?** (충동↑일 때 브레이크, Match↓일 때 비추 등)
2. **같은 점수라도 SBTI 유형이 다르면 짚는 포인트가 달라지는가?** (D형엔 감정 자극 경고, N형엔 필요성 질문 등)

이 두 가지를 동시에 검증하기 위해 **4분면 × 비교 쌍** 구조를 사용한다.

### 4분면 정의

Impulse Score와 Match Score의 조합으로 4분면을 나눈다.

| 분면 | Impulse | Match | 또바 예상 방향 |
|------|---------|-------|----------------|
| Q1 | 70+ | 65+ | 잘 맞지만 충동 요인 짚기 |
| Q2 | 70+ | 35− | 강한 브레이크 |
| Q3 | 35− | 65+ | 조건부 격려 |
| Q4 | 35− | 35− | 명확한 비추 |

### SBTI 10종 사용 원칙

**각 시나리오에서 SBTI 유형을 중복 없이 사용한다.**

같은 유형이 반복되면 "이 응답이 유형 때문인지 점수 때문인지" 구분할 수 없다.
10개 시나리오에서 10가지 유형을 모두 다르게 써서 분석력을 확보한다.

## 비교 쌍 상세

비교 쌍은 **동일 상품 + 다른 SBTI 유형**으로 구성된다.
같은 상품에 대해 두 유형의 응답을 나란히 놓고, 또바가 유형을 실제로 반영하는지 확인한다.

### Q1 쌍: S01 DUTE vs S02 NIMO

**공유 상품:** `[트렌드/FW신상] 루즈핏 레이어드 셔츠 SET` (무신사, 34% 할인, 리뷰 2100개)

| | DUTE | NIMO |
|--|------|------|
| 예상 Impulse | 75~85 | 30~42 |
| 예상 Match | 55~65 | 68~76 |
| 또바 방향 | D형 충동 자극 + E형 할인 경고 | N형 필요성 질문 + M형 대중화 경고 |

> 핵심: **같은 상품이 유형에 따라 다른 분면에 놓인다.** DUTE에겐 Q1(충동+잘맞음), NIMO에겐 Q3(충동낮음+잘맞음).

## 긴장관계 시나리오

S-BTI에는 두 가지 축 간 긴장관계가 있다. 이를 테스트하기 위한 전용 시나리오.

### S09: IT 긴장관계 — DITE

**DITE = D(충동) + I(자기판단) + T(유행참고) + E(거래이득)**

- **T형**: 유행 안에서 후보를 찾는다. 랭킹·인기 문구에 끌린다.
- **I형**: 최종 선택은 반드시 자기 기준으로 깐깐하게 검토한다.
- **긴장**: 랭킹 1위라서 눈에 들어왔는데, 내 기준에서 진짜 맞는지는 모르겠다.

**상품:** `[랭킹1위/이번시즌 필템] 와이드 데님 팬츠` (무신사, 20% 할인)

style_similarity를 **18**로 의도적으로 애매하게 설정 → 긴장이 가장 잘 드러나는 구간.

**또바가 짚어야 할 것:** "랭킹이라서 눈에 들어온 건지, 네 기준에서 직접 따져봤을 때 맞는 건지 구분해봐"

---

### S10: UM 긴장관계 — DUMO

**DUMO = D(충동) + U(타인검증) + M(차별화) + O(물건가치)**

- **U형**: 리뷰가 많을수록 믿음이 생긴다.
- **M형**: 리뷰가 너무 많으면 오히려 흥미를 잃는다.
- **긴장**: 검증은 받고 싶지만 대중화되면 싫어진다. 소수 검증 구간이 이 긴장의 최적 지점.

**상품:** `[한정수량/무신사 단독] 빈티지워시 데님 재킷` (무신사, 20% 할인, 리뷰 95개)

리뷰 **95개** = U형에겐 약간 부족, M형에겐 아직 대중화 전 → 긴장 극대화 구간.

유저가 이미 구매를 결정한 상태 → 또바는 "마음 정했다는 거 알겠어, 근거 같이 확인해볼게" 방향.

**또바가 짚어야 할 것:** "소수한테만 검증된 게 좋은 거지, 많아지면 흥미 잃을 것 알지?"

## 파일 구조

```
configs/
├── scenarios_overview.md   ← 이 파일 (전체 설계 개요)
├── scenario_01.yaml        ← S01 DUTE  (Q1-A)
├── scenario_02.yaml        ← S02 NIMO  (Q1-B)
├── scenario_03.yaml        ← S03 DUME  (Q2-A)
├── scenario_04.yaml        ← S04 NUTO  (Q2-B)
├── scenario_05.yaml        ← S05 DUTO  (Q3-A)
├── scenario_06.yaml        ← S06 NIME  (Q3-B)
├── scenario_07.yaml        ← S07 NUMO  (Q4-A)
├── scenario_08.yaml        ← S08 DIME  (Q4-B)
├── scenario_09.yaml        ← S09 DITE  (IT 긴장관계)
└── scenario_10.yaml        ← S10 DUMO  (UM 긴장관계)
```

각 YAML 파일 구조:
```yaml
id:               # 시나리오 ID (S01~S10)
label:            # 한 줄 설명
comparison_pair:  # 비교 쌍 (Q1~Q4 또는 null)
utterances:       # 발화 3종 (공통)
user:             # SBTI 유형, 나이, 스타일 태그, 후회 이력
product:          # 상품명, 플랫폼, 가격, 리뷰, 마케팅 트리거
context:          # 관심 기간, 발견 경로, 가격 체감, 접촉 이유, style_similarity
expected:         # 예상 점수 범위, 검증 체크포인트
```

## 점수 계산 방식 (참고)

Impulse Score와 Match Score는 `shared/scoring/` 모듈로 계산한다.
시나리오 YAML의 값을 그대로 `01_build_scenarios.py`에 넣으면 점수가 산출된다.

**Impulse Score에 영향을 주는 주요 변수:**
- `discount_rate`: 36~50%가 최대 자극 구간
- `review_count` + `rating`: U형에게 크게 작용
- `like_count`: M형은 임계값(플랫폼별 상이) 초과 시 감점
- `trend_hype` / `bundle` / `confidence`: 마케팅 트리거 (T형·D형 가산)

**Match Score에 영향을 주는 주요 변수:**
- `style_similarity`: 0~35점 (LLM 산출 → 시나리오에서 수동 설정)
- `interest`: 관심 기간 (2주 이상 = 20점)
- `discovery`: 발견 경로 (직접 검색 = 20점, SNS 수동 노출 = 6점)
- `price_feeling`: 가격 체감 (저렴 = 25점, 비싸요 = 3점)
